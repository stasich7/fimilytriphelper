# FamilyTripHelper Memory

## Plan Version Markdown Format
- UI publication markdown is a full plan version snapshot, not a diff or summary.
- Canonical working source is `context/georgia-trip-plan-current.md`; accepted snapshots live in `context/versions/vX.md`.
- Front matter is required and must include `trip_id`, `version_id`, and `title`; `trip_title` is recommended for the trip display title.
- `version_id` should use compact `vX` format, for example `v1`, `v2`, `v3`; each version must be unique.
- Each item must start with heading `## [stable_key] Title`.
- `stable_key` values must be unique inside one version and should be preserved across versions for the same logical item.
- Item metadata lines use `Key: value` directly after the heading; `Type` should be one of `route_option`, `day_plan`, `stay`, `transport`, `activity`, or `note`.
- Unknown item types may import but are not shown in the current version page groups, so publication files should use only supported UI types.
- Important user-visible details must be in item body, not only metadata, because UI mainly renders `title` and `bodyMarkdown`.
- Each item body must be non-empty.

## Markdown Rendering In UI
- Supported body features: paragraphs, line breaks, bullet lists with `- `, markdown links `[label](https://...)`, plain HTTPS URLs, and images `![alt](https://...)` or local root-relative images such as `![alt](/maps/day.png)`.
- Internal item links use markdown form `[label](item:stable_key)` and are resolved by the UI to the matching item in the current version.
- Route map images are generated PNG assets at 1600x1000 for Retina-friendly display; do not publish SVG route maps directly.
- Route map images must use a real map tile basemap with route overlays, not decorative invented geography, and should not duplicate the article date/title inside the image.
- Route map generation must happen one map at a time for user review.
- Route maps should use numbered chips on the image and matching inline chip PNGs in the related day text.
- If a day card has a map with numbered chips, the card body must include matching inline chip images next to the corresponding route points. Do not leave a new mapped day without inline chips.
- When using OSM tiles, set a project User-Agent, use `Referer: https://familytrip.stasich7.ru/`, cache tiles, and avoid rapid bulk requests.
- Route map basemap provider is MapTiler OpenStreetMap style. User explicitly allowed storing the MapTiler API key in the repo as an exception. Prefer Russian map labels; fall back to English if Russian is unavailable.
- Do not rely on tables, nested markdown structure, ordered lists, bold/italic rendering, headings inside body, or raw HTML.
- Version-page preview uses the first image plus a shortened text preview; full item body is visible on the item page.
- External links open in a new tab.

## Publication Workflow
- Meaningful plan updates should create a new version instead of overwriting an accepted version.
- Latest accepted trip plan version is `v31`.
- Next free trip plan version number is `v32`.
- `context/georgia-trip-plan-current.md` should mirror the latest accepted version.
- Every accepted version should be archived as a full snapshot in `context/versions/vX.md`.
- Unchanged blocks should be copied forward without rewriting.
- If a comment targets a specific `stable_key`, change only that block unless the user explicitly asks for broader changes.
- Service imports should use a full snapshot file, not a partial patch.

## Plan Content Style
- Write imported plan content naturally for family members.
- Avoid the Russian word `база`; prefer `проживание`, `размещение`, and `локация`.
- Keep links and images inside the item block they describe.
- Prefer readable markdown links over bare URLs.
- All prices in published plan content should be in USD.
- Convert GEL with planning rate `1 GEL = 0.37 USD`.
- Every price should state what it covers, such as per night, whole family, vehicle, person, room, or specific guest composition.
- Accommodation options should include source, search date, stay dates, guest composition, room count, sleeping places or room configuration, availability status, total USD price, nightly USD price, and risk notes when available.
